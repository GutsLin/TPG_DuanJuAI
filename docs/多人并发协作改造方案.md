# 多人并发协作改造方案

> 基于《调皮狗AI短剧平台v1.0.md》现状。目标：让多人同时协作同一剧目/同一集时不丢数据、不互相覆盖，并给出可分期落地的实施计划。

## 1. 现状问题清单

| # | 问题 | 位置 | 后果 |
|---|---|---|---|
| P1 | 所有更新均为「后写覆盖先写」，无版本校验 | 全部 PUT 路由（episodes/storyboards/characters/scenes/dramas） | 并发编辑丢数据（lost update） |
| P2 | `save_storyboards` 整集「先删后插」且不在事务中 | `backend/src/agents/tools/storyboard-tools.ts:188-211` | 并发拆解互相覆盖；中途失败留残缺数据 |
| P3 | 批量 upsert 接口整体提交列表 | `PUT /dramas/:id/characters`、`PUT /dramas/:id/episodes` | 多人编辑角色/集列表互相覆盖 |
| P4 | 生成类操作无互斥（同集可并发跑拆解/合成/拼接） | agent chat、compose-all、merge 路由 | 重复扣费、状态机错乱 |
| P5 | 前端无实时感知，看不到他人修改 | 全部页面（无 SSE/WebSocket） | 基于过期数据编辑，加剧 P1 |

不在本期范围：操作级实时协同编辑（OT/CRDT）、细粒度字段锁。本方案以「乐观锁 + 事务 + 互斥锁 + 变更通知」解决数据安全问题。

## 2. 改造总览（分期）

| 期 | 内容 | 解决问题 | 预估改动量 |
|---|---|---|---|
| 一期 | 乐观锁（version 字段）+ 关键写操作事务化 | P1、P2、P3 | 中 |
| 二期 | episode 级互斥锁（Redis）+ 前端冲突处理 | P4、P1 体验 | 小 |
| 三期 | 变更通知（SSE）让前端自动刷新 | P5 | 中 |

---

## 3. 一期：乐观锁 + 事务化

### 3.1 数据库变更

给并发编辑频繁的表加 `version` 列（`backend/src/db/schema.ts`）：

```ts
// episodes / storyboards / characters / scenes / dramas
version: integer('version').notNull().default(1),
```

生成迁移：

```bash
cd backend && npm run db:generate   # 产出 drizzle/0004_xxx.sql
```

迁移 SQL 只需 `ALTER TABLE ... ADD COLUMN version integer NOT NULL DEFAULT 1`，存量数据自动为 1，无需回填。

### 3.2 更新语义约定

所有单资源 PUT 改为「携带 version 的条件更新」：

```ts
const result = await db.update(schema.storyboards)
  .set({ ...fields, version: sql`${schema.storyboards.version} + 1`, updatedAt: new Date() })
  .where(and(
    eq(schema.storyboards.id, id),
    eq(schema.storyboards.version, clientVersion),
  ))
  .returning({ id: schema.storyboards.id })

if (result.length === 0) {
  // 区分「不存在」与「版本冲突」：再查一次
  return c.json({ code: 409, message: '数据已被他人修改，请刷新后重试' }, 409)
}
```

涉及路由（`backend/src/routes/`）：

- `episodes.ts`：`PUT /episodes/:id`
- `storyboards.ts`：`PUT /storyboards/:id`
- `characters.ts`：`PUT /characters/:id`
- `scenes.ts`：`PUT /scenes/:id`
- `dramas.ts`：`PUT /dramas/:id`

请求体增加 `version` 字段（必填）；GET 响应本就已返回整行，前端从详情数据里取即可。

**兼容策略**：上线初期可将 `version` 设为可选——不传则退化为现状的盲写并打警告日志；前端全部改造完成后转为必填。

### 3.3 批量 upsert 接口改造（P3）

`PUT /dramas/:id/characters`、`PUT /dramas/:id/episodes` 每项带 `id + version`：

- 有 `id` 的项按 3.2 做条件更新，版本不符的项挑出来；
- 响应返回 `{ updated: [...], conflicts: [...] }`（200，冲突项附当前库内 version），由前端提示用户逐项合并，而不是整单失败。

### 3.4 `save_storyboards` 事务化（P2）

`backend/src/agents/tools/storyboard-tools.ts` 的整集替换改为单事务，并校验 episode 版本：

```ts
await db.transaction(async (tx) => {
  // 1. 条件占位：episode version 匹配才继续，不匹配抛错回滚
  const locked = await tx.update(schema.episodes)
    .set({ version: sql`${schema.episodes.version} + 1` })
    .where(and(eq(schema.episodes.id, episodeId), eq(schema.episodes.version, expectedVersion)))
    .returning({ id: schema.episodes.id })
  if (locked.length === 0) throw new Error('EPISODE_CONFLICT')

  // 2. 删除旧分镜 + 关联，插入新分镜（原有逻辑原样搬入）
  ...
})
```

`read_storyboard_context` 返回值里带上 `episode.version`，Agent 工具入参增加 `expected_version`，由工具内部校验。冲突时工具返回结构化错误，Agent 将其呈现给用户（「分镜已被他人更新，是否覆盖/先查看」）。

### 3.5 前端配套

- `useApi` 的各 update 方法自动带上当前数据的 `version`；
- 收到 409 时 toast「内容已被他人修改」并提供「刷新数据」按钮（workbench 页调用现有 `refresh()` 即可）；
- 编辑中的表单本地保留，刷新后提示用户手动合并（不做自动三方合并）。

## 4. 二期：episode 级互斥锁

### 4.1 锁工具

复用现有 Redis（`backend/src/queue/connection.ts`），新增 `backend/src/utils/locks.ts`：

```ts
export async function withEpisodeLock<T>(episodeId: number, ttlSec: number, fn: () => Promise<T>): Promise<T> {
  const key = `lock:episode:${episodeId}`
  const ok = await redis.set(key, process.pid + ':' + Date.now(), 'EX', ttlSec, 'NX')
  if (!ok) {
    const err = new Error('该集正在执行其他生成任务，请稍后再试')
    ;(err as any).statusCode = 423 // Locked
    throw err
  }
  try {
    return await fn()
  } finally {
    await redis.del(key)
  }
}
```

### 4.2 加锁点

| 操作 | 位置 | TTL |
|---|---|---|
| storyboard_breaker 拆解 | `routes/agent.ts`（type=storyboard_breaker 时） | 10 min |
| compose-all 批量合成 | `routes/compose.ts` | 5 min（入队动作本身很快，锁只防并发触发） |
| merge 整集拼接 | `routes/merge.ts` | 5 min |
| 宫格图 split 写回分镜 | `routes/grid.ts` | 2 min |

前端对 423 响应统一 toast 提示。

### 4.3 状态机防重

补充检查（不依赖锁也能兜底）：

- `compose-all` 前若该集已有 `compose_queued/compose_processing` 状态的分镜，直接返回 409；
- `merge` 前若 `video_merges` 已有该集 `processing` 记录，返回 409。

## 5. 三期：变更通知（SSE）

> 目的：他人保存后，我的页面自动刷新，从源头上减少冲突。

### 5.1 后端

- 新增 `GET /api/v1/dramas/:id/events`（Hono `streamSSE`，需登录 + viewer 权限）；
- 进程内 `EventEmitter` 按 dramaId 分发；写操作成功后发布事件 `{ resource: 'storyboard'|'episode'|..., resourceId, action: 'updated', by: userId }`；
- 单实例部署够用；若未来 API 多实例，改走 Redis Pub/Sub（Redis 已是基础设施）。

### 5.2 前端

- `useApi` 增加 `subscribeDramaEvents(dramaId)`（EventSource 封装，自动重连、带 token 可用 query 参数或换 fetch 流）；
- workbench 页收到事件后：若 `by` 不是自己，轻量提示「xxx 更新了分镜」并静默 `refresh()` 对应区块；自己正在编辑的卡片不强制覆盖，仅标记「有更新」。

## 6. 迁移与上线步骤

1. 开发分支实现 3.1–3.4，`npm run db:generate` 产出迁移；本地 `docker compose up -d postgres redis` 验证。
2. 后端先上线（version 可选模式），观察警告日志一周，确认无第三方调用方依赖旧行为。
3. 前端上线（携带 version + 409 处理）。
4. 后端切 version 必填。
5. 二期、三期独立排期，互不强依赖。

## 7. 验收标准

- 两个会话同时编辑同一分镜：后提交者收到 409，数据与先提交者一致；
- 两个会话同时对同一集跑分镜拆解：一方成功，另一方收到冲突/423 提示，`storyboards` 与 `storyboard_characters` 数量一致无残缺；
- 一人拆解进行中，另一人触发 compose-all：收到 423；
- 杀掉 agent 进程模拟拆解中断：分镜表保持拆解前状态（事务回滚）；
- 三期后：A 保存分镜，B 的页面 3 秒内自动刷新且 B 正在编辑的内容不被覆盖。

## 8. 回归风险

- `save_storyboards` 事务化后，单事务内批量插入大分镜集耗时长，注意 PG 锁竞争；可分集测试 200+ 分镜场景。
- 条件更新改动集中在 update 语句，注意别漏 `.returning()` 判空，否则冲突会被误判为成功。
- 前端旧缓存页面（改版前打开的 tab）提交时不带 version：过渡期靠「可选 + 警告日志」兜底，强制切换前公告。
